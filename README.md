## 🟢프로젝트 설명
기후위기가 날로 심각해지면서 환경보호에 대한 대중의 관심사가 커져가고 있지만, 개인이 생활 속에서 실천할 수 있는 것이 무엇인지는 명확하게 제시되지 못하고 있다. 이러한 문제의식을 가지고 진행한 프로젝트

고객의 가정, 사무실에서 배출된 폐기물을 수거하여 업사이클링 제품 제작 업체에 전달.

업사이클링 업체가 제작한 제품을 게시하고, 고객에게 판매 및 배송.

일련의 과정을 매개하는 업사이클링 플랫폼.

## 🟢빌드 및 실행방법
웹 앱 기반 프로젝트로, `아이폰 사파리, 갤럭시 크롬, 삼성 인터넷` 3가지의 경우에서 ‘웹 앱‘ 에 최적화되어 홈 화면에 바로가기 앱 아이콘을 추가할 수 있습니다.

실행 주소: [올:리본](https://dev-allreborn.netlify.app)

## 🟢정리

### 🌱개발 전 목표/고려사항
- TypeScript 의 적용
- 컴포넌트의 분리 및 재사용성 고도화
- 폴더 구조/구분
- PWA
- naver maps API 활용
- 서비스 오픈

### 🌱배우고 느낀 점
- PWA 기반을 고려하고 개발을 진행했는데, 웹 앱으로 각 브라우저(사파리, 삼성, 갤럭시 크롬)마다 지원하는 일부 고유 UI 가 달라서 컴포넌트의 디자인이 일치하지 않는 문제를 겪었다. 하나의 언어로 여러가지 플랫폼을 수용할 수 있어서 좋은 점도 있었지만, 개발자 혹은 기획자가 예상한 사용 경험을 주려면 각 브라우저에 대한 대처도 필요하겠다는 생각이 들었다. 하지만 웹 기반의 페이지들에게는 추가적인 리소스(개발인력, 새로운 언어)를 투자하지 않아도 된다는 분명한 이점이 있다고 생각한다!
- 서비스워커를 이용하면 웹앱을 보다 더 "앱처럼" 활용할 수 있게 해준다. (데이터를 캐싱해놓는다던지, 바로가기 앱 설치 알람을 띄워준다던지) 하지만 생각보다 적용시키는게 어려웠다. 이것저것 적용시켜보다 지속적인 충돌을 발생시켜서 이번 프로젝트에서는 우선 제외 시켰으나, 추가적으로 적용시켜보자.
- 타입스크립트의 타입 선언 후 타입 추론, 타입 추천은 정말 강력한 기능이라고 생각한다! 처음에는 코드 작성자는 타입 선언이라는 일을 추가로 해야하니 오히려 불편하지 않을까 생각했다. 하지만 컴포넌트를 생성할 때 props 가 정의되어 있는점, 반드시 필요한 값을 에러로 확인하지 않고 코드 작성동안에 해결할 수 있다는 점이 매우 편했다. 그럼에도 내가 생각하는 핵심은 타입스크립트는 동료를 위해 필요하다고 생각한다. 내가 작성하지 않은 코드는 얼마나 더 복잡할까? 그 힘듦을 조금이라도 줄여주는 것이 타입스크립트라는 생각이 들었다.
- 조금 더 깔끔한 폴더/파일 분리와 재사용성이 높은 컴포넌트를 만들기 위해 고민했던 것같다. 큰 프로젝트는 아니었지만 버튼컴포넌트 등 공용 컴포넌트를 만들면서 재미있었다.
- manifest 파일을 이용하여 기기에서 페이지가 앱처럼 보여질 때 기타 스타일을 설정하여 웹앱을 간단하게 구현할 수 있다!

### 🌱아쉬운 점
1. 처음에는 백엔드 개발자분까지 합류하여 서비스 오픈을 목표로 개발할 예정이었으나, 다수의 사이드 프로젝트가 그러하듯 팀원들의 하차...로인해 끝까지 끌고가지 못한 것이 아쉽다. 아쉽지만 혼자라도 1차 구현목표까지 완료했다.
2. naver maps API 를 이용해서 최초 기획보다 더 다양한 기능을 구현하고 싶었다. 내위치 리셋, 핀 클릭 시 상세조회... 등. 그런데 웹에서는 정상적으로 동작되는 것들이 모바일에서는 정상적으로 적용이 안되는 경우가 있었다. 지도를 이용해서 더 다양하고 재미있는 기능들을 정상적으로 구현할 수 있었다면 하는 아쉬움이 있다... 계속 공부하고 시도해봐야겠다..
3. serviceWorker 적용이 잘 안됐다.
